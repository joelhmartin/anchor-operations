# Operations — Architecture

**Last updated:** 2026-06-25 (client-first redesign — Tasks 1–9)

Operations is the multi-platform health-check + AI command center for client
properties. It runs scheduled and on-demand "runs" across three umbrellas
(website, Google Ads, Meta), persists findings, correlates anomalies across
platforms, emails digests, and exposes a per-client AI supervisor with
read/mutate-with-approval tools.

This document is the authoritative architecture overview. It supersedes the
older `docs/superpowers/plans/2026-04-29-kinsta-operations.md` (now archived)
and the per-phase docs in `docs/ops-rebuild/phases/`.

---

## 1. Overview

| Surface | What it does |
|---|---|
| Run executor | Materializes a run from a `run_definition`, fans out to checks, persists `ops_check_results` + `ops_findings` |
| Correlator | Cross-platform rule engine that emits umbrella-level findings (e.g. "ad spend up + form submits down") |
| Reports | In-app HTML reports (PDF deferred per P4) summarizing a run window |
| Email digest | Mailgun-driven roll-up of new critical findings, opt-in per client subscription |
| AI supervisor | Per-client chat that delegates to website / google_ads / meta sub-agents; mutations require admin approval |
| UI | Left-rail workspace at `/operations` — three views: **Home** (curated digest), **Clients** (roster + per-client page), **Portfolio** (bulk & cost). Per-client page has primary tabs Overview · Findings · Socials · Blog · Sites · Chat and a Config menu with Health checks · Connections · Run history · Cost. |

Built across phases 0–10 of the Operations rebuild (May 2026). Locked
decisions live in `docs/ops-rebuild/STATE.md`.

## 2. Domain model

Schema lives in `server/sql/migrate_ops_*.sql` (foundation in
`migrate_ops_foundation.sql`).

| Table | Purpose |
|---|---|
| `ops_run_definitions` | Reusable check-set bundles (e.g. `web_daily_essential`, `gads_weekly`). Owned by run definitions seeded in migrations + admin-editable via Schedule tab. |
| `client_run_subscriptions` | Many-to-many between clients and run definitions. Carries cadence + active flag. |
| `ops_runs` | One row per run instance (`queued → running → completed/partial/failed/cancelled/budget_exceeded`). Holds `started_at`, `finished_at`, `token_usage_json`, `cost_cents`. |
| `ops_check_results` | One row per check executed in a run. Status (`pass/fail/warn/skipped/error`), payload JSON (sanitized), duration. |
| `ops_findings` | Cross-run actionable items derived from check_results + correlator. Severity (`info/warning/critical`), `acknowledged_at`, `resolved_at`, `client_user_id`, `umbrella`. |
| `ops_reports` | Persisted run reports (HTML) generated by `reportRenderer.js`. |
| `ops_tool_approvals` | Durable approval audit for AI mutations. `tool_proposed → approved/rejected → executed`. |
| `client_platform_credentials` | Encrypted per-client credentials for ad platforms (Phase 4/5/8 OAuth + system tokens). AES-256 at rest. |
| `ops_vuln_feed` | Mirrored WPVuln feed used by `web.wp_security`. |
| `kinsta_findings` | Legacy drift findings — kept for compat (still written by `web.kinsta.drift`); read also exposed via `kinsta_findings_compat` view. New findings flow through `ops_findings`. |
| `audit_runs` / `audit_schedules` | Legacy analytics audits — kept separate per P6. Do not extend. |

Compat view: `kinsta_findings_compat` (Phase 1) maps legacy drift findings
into the `ops_findings` shape so a single query joins both sources during
the deprecation window.

## 3. Run lifecycle

States: `queued → running → completed / partial / failed / cancelled / budget_exceeded`.

```
schedule fanout (cron)         manual trigger (UI / Re-run)
        │                            │
        ▼                            ▼
   ops_runs (queued)
        │
        ▼
   runQueue.js → runExecutor.js
        │
        ├── checks/registry.js → fan out to per-umbrella check modules
        │       └─ each check writes ops_check_results
        │
        ├── correlator.js (post-fanout) → emits ops_findings
        │
        ├── reportRenderer.js → ops_reports (HTML)
        │
        ├── emailDigest.js (if subscribed) → Mailgun
        │
        └── budgetGuard.js trips → status = budget_exceeded
```

`runExecutor.js` applies `payloadSanitizer.js` to every check payload before
persistence (PHI scrub: emails, phones, DOBs are redacted unless the column
key is non-user-ish).

## 4. Check registry + umbrellas

Three umbrellas: `website`, `google_ads`, `meta`.

- Registry: `server/services/ops/checks/registry.js`
- Per-umbrella modules:
  - `server/services/ops/checks/website/` — Kinsta drift, PSI, SSL, schema, GSC, SEMrush, WP security, broken-links stub, tracking install
  - `server/services/ops/checks/google_ads/` — disapprovals, conversion tracking, spend deltas, keyword history (gRPC via `google-ads-api`)
  - `server/services/ops/checks/meta/` — pixel events, ad delivery, audience size **gated by HIPAA** (`client_type !== 'medical'`); medical clients short-circuit to `skipped` with neutral reason
- Wrappers: `web.kinsta.drift` is a Phase-3 wrapper around the legacy
  drift scanner so the new umbrella speaks one shape. Source code:
  `server/services/ops/checks/website/kinstaDrift.js` calls into
  `server/services/ops/operations-website/driftScanner.js`.

## 5. Orchestration

Two execution paths:

1. **Local / dev (in-process):** `runQueue.js` works directly off the DB.
   `scheduleFanout.js` runs a cron tick that materializes due
   subscriptions into `ops_runs (queued)` rows. The same process picks
   them up via `runExecutor.js`.
2. **Production:** The same `scheduleFanout.js` enqueues to Pub/Sub; a
   Cloud Run Job consumes and runs `runExecutor.js`. Cloud Scheduler
   triggers the fanout HTTP endpoint on the public service.

Run definitions ship per-tier (daily essentials / weekly deep / monthly
audit per P8). Seed migrations:
`migrate_ops_seed_run_definitions.sql`, `_meta_`, `_gads_`.

---

## 6. Cost tracking + monthly cap

Per P7, every client has a configurable monthly cap (default $5/month).

- `costTracker.js` accumulates Vertex token cost per run + per-subagent.
- `client_profiles.ops_monthly_cap_cents` (column added in
  `migrate_ops_monthly_cap.sql`) overrides the default.
- `budgetGuard.js` checks MTD spend before each costly hop. When the cap
  trips mid-run, `ops_runs.status = 'budget_exceeded'` and remaining
  hops short-circuit.
- Cost UI: Operations → Cost tab shows MTD spend per client + by tier
  + by sub-agent, with an "Edit cap" dialog backed by
  `PUT /api/ops/clients/:id/cap`.

---

## 7. Cross-platform correlator + reports + email

- `correlator.js` runs after the per-check fanout. Reads recent
  `ops_check_results` for the run's client and emits umbrella-level or
  cross-umbrella `ops_findings`.
- Rules live in `correlatorRules.js` (declarative). Examples: spend
  spike + conversion drop, PSI regression coincident with theme update.
- `reportRenderer.js` writes an HTML report into `ops_reports`. P4 locked
  HTML-only; PDF deferred.
- `emailDigest.js` rolls up new critical findings since last digest and
  sends via Mailgun. Subscriptions opt-in per client (Phase 8 column
  added in `migrate_ops_subscription_email.sql`).

## 8. AI supervisor + sub-agents

Replaces the legacy single-agent assistant (Phase 7).

Architecture:

```
src/views/admin/Operations/Chat/ClientChat.jsx
        │ POST /api/ops/chat
        ▼
server/routes/ops.js → runSupervisorTurn (server/services/ops/agents/supervisor.js)
                                │
                                ├── load_run / drill_into
                                ├── delegate_to → server/services/ops/agents/subAgents/<name>Agent.js
                                │                        └─ runSubAgentLoop (read-only during delegate)
                                └── propose_action → ops_tool_approvals (pending)
                                                          │
                                                          ▼
                              POST /api/ops/chat/approve → executeApproval → subagent tool handler
                                                          │
                                                          ▼
                                  audit: operations.tool_approved + .tool_executed
```

Sub-agents:

- `subAgents/websiteAgent.js` + `websiteTools.js` — Kinsta WP-CLI read,
  SFTP read, plugin list, posts list, PSI on demand, GSC query, SEMrush
  lookup, tracking-install verification. Mutators (`plugin_update`,
  `wp_user_password_reset`) require approval.
- `subAgents/googleAdsAgent.js` — read-only `gads_query` (SELECT-only
  GAQL with `LIMIT 200` cap), keyword history, disapproval reason.
  Mutations deferred per P5.
- `subAgents/metaAgent.js` — `meta_query`, `meta_pixel_test_event`. HIPAA
  gate at agent entry — medical clients get a neutral refusal and zero
  Vertex spend.

Budget: `vertexRuntime.js` enforces a 50¢ per-turn hard cap shared
between supervisor + delegated sub-agent calls. Monthly cap is enforced
by `budgetGuard.js` over the per-client `cost_cents` aggregation.

Approval audit chain (every mutation):
- `operations.tool_proposed` (when supervisor calls `propose_action`)
- `operations.tool_approved` + `operations.tool_executed` (on approve)
- `operations.tool_rejected` (on reject)

All events route to `SecurityEventCategories.OPERATIONS`.

## 9. UI — client-first left-rail IA

`src/views/admin/Operations/index.jsx` (renamed from `OperationsWorkspace/` in
Phase 10). Redesigned in the client-first pivot (Tasks 1–9, June 2026).

### 9.1 Shell

A 180 px left rail (`WorkspaceRail.jsx`) offers three top-level views. State is
driven by query params so URLs are deep-linkable and bookmarkable:
`?view=<home|clients|portfolio>&clientUserId=<uuid>&section=<section>`.

Context provider: `OpsWorkspaceContext.jsx` — loads the client roster on mount
and derives `activeClient` from `clientUserId`. The `setClientUserId` helper
always switches `view → clients` and `section → overview`.

### 9.2 Home view

`home/HomeDigest.jsx` — curated digest backed by `GET /api/ops/home`.

| Card | Content | Deep-link on click |
|---|---|---|
| Needs Attention | Clients with unresolved critical findings; count badge + top finding snippet | Opens client → Findings section |
| Scheduled Today | Blog posts and social posts publishing today | Opens client → Blog or Socials section |
| Approvals Waiting | Count of pending `ops_tool_approvals` rows across all clients | Prompt to open a client's Chat section |

### 9.3 Clients view

A `Clients/ClientRoster.jsx` sidebar lists all ops-enrolled clients with a
status dot (red = any open critical finding). Selecting a client navigates to
`Clients/ClientWorkspace.jsx` which renders:

**Primary tabs** (always visible):

| Section | Component | Purpose |
|---|---|---|
| Overview | `Clients/ClientOverview.jsx` | 4 stat cards (open findings, posts scheduled, MTD spend, cap) + top 5 notable findings + content scheduled in next 48h. Backed by `GET /api/ops/clients/:id/overview`. |
| Findings | `Discoveries/DiscoveriesTab.jsx` | Cross-run finding feed, ack/resolve, bulk-ack — filtered to the active client |
| Socials | `Content/ContentTab.jsx` (mode=social) | Per-client social post queue |
| Blog | `Content/ContentTab.jsx` (mode=blog) | Per-client blog queue |
| Sites | `Clients/ClientSitesPanel.jsx` | Per-client Kinsta site list with assign picker. Revives the orphaned Sites feature using legacy `/api/operations/sites*` endpoints (no new route needed). |
| Chat | `Chat/ClientChat.jsx` | Per-client AI supervisor turn loop |

**Config menu** (gear button, rendered in `ClientOpsView.jsx`):

| Item | Purpose |
|---|---|
| Health checks | Subscription editor (run definitions + cadence) |
| Connections | Per-client credential health |
| Run history | Filtered run list + Re-run + report download |
| Cost | MTD spend with cap chip + "Edit cap" dialog |

### 9.4 Portfolio view

`portfolio/PortfolioView.jsx` — agency-wide cross-client surfaces:

- `Bulk/BulkTab` — run an action against many environments with bounded concurrency
- `Cost/CostTab` — per-client MTD spend by tier/sub-agent + "Edit cap"

### 9.5 New API endpoints (client-first redesign)

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/ops/home` | Home digest — needs-attention clients, content scheduled today, pending approvals count. Internally delegates to the same `loadCommandCenter()` helper used by `/api/ops/command-center` (unchanged). |
| `GET` | `/api/ops/clients/:id/overview` | Per-client digest for the Overview section — stat cards, top findings, upcoming content. |

### 9.6 Orphaned pre-pivot folders

The following folders were created during earlier phases and are no longer
wired into the shell. They are **not deleted** (out of scope) but should not

`Overview/`, `Connections/`, `Runs/`, `Schedule/`, `Cost/` (old tab components),
`CommandCenter/`, `Discoveries/` (old top-level views), `Sites/` (old top-level
Sites tab replaced by per-client `Clients/ClientSitesPanel.jsx`).

## 10. Compliance posture

Operations handles PHI via the `client_profiles.client_type` field —
clients flagged `medical` are HIPAA-protected.

- **HIPAA gate on Meta:** the Meta sub-agent and every Meta check
  refuse to execute for `client_type='medical'`. Defense in depth at
  agent entry + per-tool. Refusal text is neutral ("Meta tooling is not
  available for this client") — `client_type` is never echoed.
- **PHI sanitizer:** `payloadSanitizer.js` scrubs emails / phones / DOBs
  from check payloads before they hit `ops_check_results.payload_json`.
  Logged events never include PHI.
- **Approval audit:** Every mutation has a durable row in
  `ops_tool_approvals` plus four security events (proposed, approved,
  executed, rejected). Audit rows are immutable once `executed_at` is
  set.
- **Credential storage:** `client_platform_credentials.encrypted_blob`
  is AES-256 (`server/services/security/encryption.js`).
- **Credential deletion audit:** `DELETE /api/ops/clients/:id/credentials/:credentialId`
  emits a `operations.credential_deleted` security event
  (`SecurityEventTypes.OPERATIONS_CREDENTIAL_DELETED`,
  `SecurityEventCategories.OPERATIONS`) with `clientUserId` + `credentialId` in
  `details`. Added in the client-first redesign.
- **Authz gaps closed (client-first redesign):** `PUT /clients/:id/subscriptions`,
  `PUT /clients/:id/credentials/:platform`, and
  `DELETE /clients/:id/credentials/:credentialId` now all gate on
  `isOperationsClient(req.params.id)` before executing. Previously these endpoints
  accepted any valid UUID without verifying the client was enrolled in Operations.
- **SSRF guard:** `verify_tracking_install`, `psi_run_now`, and any
  outbound HTTP from the agent stack route through
  `server/services/security/ssrfGuard.js` (private-IP block + DNS
  pinning).
- **Rate limits:** `/api/ops/chat`, `/api/ops/runs/:id/...` and the
  legacy `/api/operations/exec` use the per-user policies registered in
  `server/services/security/rateLimit.js`.
- **WP-CLI allowlist:** Tokenized parser in
  `server/services/ops/operations-website/agentTools.js` (and mirrored
  in the new sub-agent tool module) restricts WP-CLI to a known-good
  read-only set.
